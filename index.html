关于CSS选择器

选择器类型

基础选择器 (Basic Selectors)

  • 元素选择器

    通过Node节点匹配元素,在单独使用时,寻找特定类型的元素时,元素选择器都会匹配该文档中所有此类型的元素.

    元素 {样式声明}

  • 类选择器

    CSS类选择器会根据元素的类属性中的内容匹配元素，需要完全匹配，以空白分隔

    .类名 {样式声明}

    类选择器等同于下述：

    [class ~= 类名] {样式声明}

  • ID选择器

    根据元素中ID属性中的内容匹配元素，需要完全匹配

    #ID {样式声明}

    ID选择器等同于下述:

    [id = id属性的值] {样式声明}

  • 通配选择器

    可以匹配任何HTML类型的元素，故可省略。在CSS3中，较经典的与命名空间搭配使用

    • a|*会匹配a空间下的所有元素
    • |*会匹配所有无命名空间的元素

  • 属性选择器

    根据属性中的内容匹配元素，需要属性值完全匹配

    语法

    [attr]
    表示带有以attr命名的属性的所有元素
    [attr=value]
    表示带有以attr命名的属性且值等于value的所有元素
    [attr~=value]
    表示带有以attr命名的属性，value是以空白分隔的值列表，匹配HTML中该属性的值是值列表之一的元素，例如class,className(类属性选择器)
    [attr|=value]
    表示带有以attr命名的属性，value是value或者以value-为前缀的值，例如匹配zh-CN,zh-TW,其中zh可以作为上述value
    [attr^=value]
    表示带有以attr命名的属性，value是value为前缀的值，例如匹配zh-CN,zh-TW,其中zh可以作为上述value
    [attr$=value]
    表示带有以attr命名的属性，value是value为前缀的值，例如匹配abc,acbc,其中bc可以作为上述value
    [attr$=value]
    表示带有以attr命名的属性，value是value为后缀的值，例如匹配abc,acbc,其中bc可以作为上述value
    [attr*=value]
    表示带有以attr命名的属性，value被包含于html中填写的实际的值，例如匹配abc,acbc,其中bc可以作为上述value
    [attroperatevalue]
    在属性选择器的右方括号前添加一个用空格隔开的字母 i（或 I），可以在匹配属性值时忽略大小写

复合选择器 (Combinators)

  • 相邻兄弟选择器

    介于两个选择器之间，需要两个元素同属一个父元素且相邻

    strong + span { color: red; }

    Result:

    1 2 3 4 5

  • 通用兄弟选择器

    位置无须紧邻，只须同层级，A~B选择A元素之后所有同层级B元素。

    em ~ span { color: blue; }

    Result:

    1 2 3 4 5

  • 子选择器

    匹配第一个元素的子元素，孙元素及以后无法匹配

    #show_child_selector div > span { color: green }

    Result:

    1 2 3

    注意此处若没有给二级span加样式则会继承一级span的样式

伪类 (Pseudo Classes)

  • 常用的伪类

    :link
    匹配未被访问，且有href属性的a标签
    :visited
    匹配已被访问的链接
    :hover
    匹配光标所在的元素
    :active
    匹配已操作过，比如按钮按下过鼠标左键等
    :first-child
    表示一组同级元素中的第一个元素
    :first-of-type
    表示该标签下面的第一个标签元素
    :last-child
    表示一组同级元素中的最后一个元素
    :last-of-type
    表示该标签下面的最后一个标签元素
    :not
    表示与选择器列表不匹配的元素
    :target
    通过a标签的href属性与之对应id元素匹配

    Result:

    test1 test2 test3

    test4test5

    test6

    注意：这里面first-child与first-of-type优先级一样所以被覆盖了

伪元素 (Pseudo Elements)

  • 常用的伪元素

    ::before
    在选中的元素前添加样式声明
    ::after
    在选中的元素后添加样式声明
    • Buy milk
    • Take the dog for a walk
    • Exercise
    • Write code
    • Play music
    • Relax

CSS选择器的优先级

计算优先级

CSS是通过进位制计算的优先级，(ID，类，标签),为了好做计算，分别将其对应为（a,b,c）

  1. a是id选择器的次数
  2. b是类选择器，属性选择器（排除ID选择器和标签选择器），伪类的次数
  3. c是标签选择器以及伪元素的次数

当前选择器的优先级权重 = a * 100 + b * 10 + c * 1

此刻选择器的优先级比较已经出现了，权重值大的优先，权重值相等后声明的优先。

当然，还有其他两个我个人很不推荐的影响优先级的方法。其中一个是内联，一个是!important,如果当前选择器有内联样式，则其内联样式声明优先级比选择器里面的样式声明要高，而其中有!important的样式优先级最高。从之前也可以推出内联里面的！important比选择器里的！important优先级高

JavaScript的异步机制

Asynchronous Callbacks

经典的callback(回调)模式存在层级嵌套的问题，如果你对node.js有一定了解一定会听过所谓的callback-hell(回调地狱）,增加了运行时间以及嵌套层级。是此前JavaScript唯一的异步处理方法。

Promise

promise则通过一种链式表达的方式代替了回调的方式。可在运行异步操作并等待其完全完成后再根据其结果运行另一项操作

async/await

异步函数由async关键字定义。在一个异步函数中可以使用await关键字。async和await关键字可以使得对有等待时间的（异步），以Promise为基础的函数的定义更加简洁优雅，减少特意配置对于Promise的链式调用。

技术分享：1.自制一个迷你版浏览器(原理科普篇)

发表于 2020-09-15

一个活跃在各个前端社区的经典问题：从输入URL到页面展示其间发生了什么？

首先，准确说来应该是URI，URI通常是由 协议名 + 主机名 + 路径 + 查询参数 构成。
URI涉及到了一个知识体系，既域名的概念，域名是IP地址的等价替代，是一个树状的分布式查询系统，通过多级缓存提高查询效率。（CDN运用中客户端最常用的数据存在二级缓存，二级缓存没有再访问一级缓存，一级缓存没有才溯源。其中涉及到一个命中率和回源率的术语）。URI其中的协议名我目前接触到的http,https,以及ws(wss)。

1.http想必大家已经耳熟能详了，一个建立在TCP/IP协议之上的超文本传输协议。（无状态，应用层，以请求应答方式运行的协议，通过可扩展语义和自描述消息格式，以基于超文本信息系统灵活互动）。其中1.1开始默认进行长连接（节省多次握手挥手的开支）,HTTP/2,HTTP/3新增了Stream, Sever Push等特性，解决了队头堵塞等问题，其中HTTP/3不再支持明文传输。
2.https粗略点说就是http + TCP/IP + SSL/TLS, 其中SSL/TLS涉及到了一个概念既混合加密，通过非对称加密 加密 对称加密 的公钥来提高其安全性。
3.ws,wss分别代表明文和加密的websocket协议。采用的也是如同HTTP/2后的二进制帧，不过却没有流（Stream）的概念，也就不存在多路复用，优先级等复杂特性，其自身是“全双工”的。websocker基于Http之上，是一个标准的get请求（Http方法:get,head,post,put,delete,connect,options,trace), 通过“Connection:Upgrade”以及“Upgrade:websocket”来识别。为避免误认还多了“Sec-WebSocket-Key”以及“Sec-WebSocket-Version”。不支持#，得用%23代替。

回到正题，在输入URI后，
一：首先是DNS的解析，其中解析顺序是：
1.浏览器缓存 -> 2.操作系统缓存 -> 3.hosts文件 -> 4.非权威域名服务器(大公司自建) -> 5. 根域名服务器 -> 6.顶级域名服务器 ->7.二级域名服务器 -> 8.权威域名服务器
如果缓存中有则直接通过域名对应的IP地址,反之则最后解析出来得到一个IP地址,总的来说DNS只不过是个命名空间。其中可能还有CDN（内容分发网络）的中转处理,如果CDN中缓存中有着对应的则直接返回,反之则回源到服务器。经过防火墙外网网管路由到nginx接入层,有则返回，没有则负载到对应web服务器,有则返回并写入缓存,无则经典报错404。
二：在有了IP后则可以进行下一步的第三层传输层的TCP协议，其中涉及到的三次握手SYN,SYN-ACK,ACK，以及结束后的四次挥手。
Client:[ESTAB-LISHED,FIN-WAIT-1,FIN-WAIT-2,TIME-WAIT,CLOSED]
Serve:[ESTAB-LISHED,CLOSE-WAIT,LAST-ACT,CLOSED]
三：如果是HTTP协议则直接到四，如果是HTTPS协议，则其中还涉及到了SSL/TLS（混合加密，数字签名和证书）,如果是websocket协议也快进到四。
四：客户端（浏览器）在接收到服务器返回的数据后渲染页面。其中渲染页面涉及到浏览器机制的问题。那么我们便再从浏览器的视角从输入URL到页面展示，这期间经历了什么？
因为我平时都采用Chrome进行开发，那么便从Chrome的角度来分析了。

在Chrome中主要有4个进程：
·浏览器进程 负责浏览器工具栏，前进后退输入，以及输入后的网络请求和文件访问。
1.UI thread: 控制浏览器工具栏操作
2.network thread: 网络请求相关操作
3.storage thread 访问文件
·渲染进程 负责对响应数据的渲染
·插件进程 各种插件，不赘述了
·GPU进程 对整个应用程序中GPU任务的处理

Step 1:UI thread 会判断输入的是否是URL，不是则跳转到google搜索
Step 2:对于浏览器上工具栏的各个按钮（前进，后退，收藏夹等）以及地址栏按回车，便会开始调用network thread, 然后就是之前所描述的从DNS解析开始…
Step 3:network thread收到服务器的响应后，根据Content-Type字段来判断是渲染还是下载文件。
Step 4:如果是html文件则交给render process
·构建DOM：Main thread解析响应数据转化为DOM对象，为了提升效率，在构建DOM的过程中，图片，CSS，JS文件等这些需要通过网络请求得到的会进行预加载处理。如果遇到<script>标签，渲染引擎则会停滞对HTML的解析跑去加载JS代码，因为JS代码可能会改变DOM结构，总的来说重绘是要比回流代价要小得多。DOM树构建，布局计算各个节点所处位置，在样式计算后当然少不了绘制啦。通过各种选择器，给对应节点加上对应样式，当然，即便你不添加样式，浏览器也会有默认样式，比如字体大小等。随后就是合成了，合成后便是所谓的页面展示了。
